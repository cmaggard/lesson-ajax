};

/* Prepares to send a request:

jQuery

  • Created by John Resig in 2006
  • JS library for interacting with contents of a web page
  • Whole bunch of convenience methods
  • Used by basically every web page/app in the world
  • $ as a convenience variable

jQuery.ajax


$.ajax({
  // Relative path to where we're getting data
  url: "/api/cats",
  
  // Executes this function when data loads
  success: function(data) {
    console.log(data);
  }
});

Now with robustness


$.ajax({
  // Relative path to where we're getting data
  url: "/api/cats",
  
  // JS object, contents passed to the server with the request
  data: { color: "brown" },

  success: function(data) {
    console.log(data);
  },
  
  // Executes this function when something breaks
  error: function() {
    doSomethingElseInstead();
  },
  
  // Executes this function after success or failure
  complete: function() {
    ayoImDoneWithMyTransaction();
  }
});

With great power comes

Limitations

No cross-domain AJAX requests

Workarounds:

  • JSONP (server config required)
  • Cross-Origin Resource Sharing (same)
  • Yahoo YQL as middleware (dirty, dirty hack)

Be careful with your code

  • No page refresh = no tolerance for memory leaks
  • It is really, really easy to write shitty Javascript

Asynchronous code is "fun"

  • Implicit multithreading makes control flow a big deal
  • Race conditions suck to debug
  • Again: It is really, really easy to write shitty Javascript

Client-server limitations

  • Request-only; servers can't push updates to clients
  • Each request is an HTTP request/response
    Transmission time + processing time can scale out of whack quickly
  • ProTip: There's a bleeding-edge way around this... look up "WebSockets"
